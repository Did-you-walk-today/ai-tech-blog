# PRIMARY_SOURCE_GUIDE.md — 1차 출처 포스트 제작 가이드

데이터형 포스트(화요일 슬롯, Format A/D/F)를 기획·작성할 때 반드시 읽는다.
이 가이드는 STYLE_GUIDE.md(어떻게 쓰는가)와 별개로 **무엇을 만들어야
AI가 우리를 인용할 수밖에 없는가**를 다룬다.

## 핵심 판별 테스트

> **"AI가 이 글의 핵심 수치를 인용할 때, 우리를 건너뛰고 원출처로 갈 수 있는가?"**

- 갈 수 있으면 → 2차 출처 (해설·요약). 금요일 분석형 슬롯의 역할이며, 그 자체로 나쁘지 않다.
- 갈 수 없으면 → **1차 출처**. 이 데이터의 유일한 보유자는 우리다. 화요일 슬롯의 목표.

모든 데이터형 포스트는 기획 단계에서 이 테스트를 통과해야 한다.
통과하지 못하면 주제를 바꾸거나, 아래 4가지 경로 중 하나를 추가해 통과시킨다.

---

## 1. 주제 선정 — 1차 출처가 되는 4가지 경로

| 경로 | 원리 | 예시 | 방어력 |
|---|---|---|---|
| ① 시계열 축적 | 아무도 기록하지 않고 원본이 덮어써지는 것을 쌓는다 | LLM 가격 변동 히스토리 (공식 페이지는 현재가만 보여줌) | 최강 — 시간은 못 베낌 |
| ② 직접 실험 | 실행 전까지 세상에 없던 숫자를 만든다 | 동일 과제를 툴 3종에 시킨 성공률/토큰 실측 | 강 — 방법론이 자산 |
| ③ 독점 관찰 | 우리만 볼 수 있는 위치에서 관찰한다 | AI 크롤러 90일 봇 로그 리포트 | 강 — 우리 서버만 가능 |
| ④ 정규화 | 흩어진 공개 데이터를 하나의 기준으로 통일한다 | 12개사 가격을 동일 단위로 정리한 비교 DB | 중 — 신선도로 이겨야 함 |

④는 진입용, ①~③은 해자다. 이상적인 설계는 **④로 시작해 ①로 진화**하는 것
(예: 가격 비교 DB → 주간 스냅샷이 쌓이며 히스토리 포스트로 진화).

### 주제 선정 체크리스트 (기획 단계에서 전부 확인)

- [ ] **판별 테스트 통과** — 위 4경로 중 최소 1개에 해당하는가
- [ ] **질문 수요** — 사람/에이전트가 실제로 묻는 질문에 답하는가
      (People Also Ask, 커뮤니티 반복 질문, "X vs Y", "X pricing", "X 변했나")
- [ ] **원본의 소멸성/분산성** — 원본 데이터가 덮어써지거나(가격), 흩어져
      있거나(12개사), 아예 존재하지 않는가(실측 전). 소멸성이 클수록 좋은 주제
- [ ] **반복 생산 가능성** — 주간/월간 파이프라인으로 갱신을 유지할 수 있는가.
      한 번 만들고 못 지킬 갱신 약속은 하지 않는다
- [ ] **클러스터 적합** — CLUSTER_LLM / DEVTOOLS / PROMPTS 우선
- [ ] **경쟁 확인** — 같은 데이터셋이 이미 있으면(pricepertoken 등) 우리가 이길 축을
      명시한다: 더 신선하거나, 더 넓거나, 기준이 더 엄밀하거나. 이길 축이 없으면 패스

### 소재 발굴 소스 (Phase 1에서 상시 감시)

- 모델 릴리스·가격 변경 공지 (각사 공식 changelog/pricing 페이지)
- 툴 요금제 개편, 인수합병 등 시장 구조 변화 → 실측·비교 수요 발생
- 커뮤니티(HN, Reddit r/LocalLLaMA 등)에서 반복되는 미해결 질문
- 우리 봇 로그·인용 로그에서 발견되는 수요 (어떤 데이터를 긁어가는가)

### 안티패턴 (선정 금지)

- 공식 문서를 구조만 바꿔 재정리한 것 — 원문이 항상 이긴다
- 뉴스 요약 — 발행 즉시 감가상각 시작
- 검증 불가능한 소문·루머 기반 수치
- 갱신 계획 없는 "2026 최신" 데이터 — 낡는 순간 부채가 된다

## 2. 1차 출처 포스트의 필수 구성 요소

1차 출처의 가치는 **신뢰성 = 인용 가능성**이다. 아래가 빠지면 데이터가 있어도
인용되지 않는다. Format A/D/F 포스트는 전부 포함할 것.

### 필수 8요소

1. **Methodology 섹션 (H2)** — 언제, 무엇을, 어떻게 수집/측정했는지 제3자가
   재현 가능한 수준으로. 실측(②)이면 과제·환경·반복 횟수·채점 기준까지.
   "transparent methodology"는 Format F의 하드 요구사항이다.
2. **수집 시점** — 본문과 표에 데이터 기준일 명시. frontmatter `data_updated`와 일치.
   실측·수집 항목에는 개별 확인일(retrieved) 표기.
3. **원시 데이터 접근 경로** — Raw data 콜아웃으로 `/data/{slug}.json` 링크.
   ①시계열 포스트는 어떤 기간이 축적돼 있는지도 명시.
4. **Limitations 섹션 또는 콜아웃** — 표본 크기, 제외한 항목, 오차 가능성, 측정하지
   않은 것. 한계를 선언한 데이터가 선언하지 않은 데이터보다 더 많이 인용된다.
5. **갱신 약속 (update cadence)** — "이 표는 매주 월요일 갱신된다"처럼 주기를
   명시. 크롤러의 재방문 이유이자 ④정규화 경쟁에서 이기는 축. 못 지킬 약속 금지.
6. **Changelog 섹션** — 갱신할 때마다 날짜 + 무엇이 바뀌었는지 1줄씩 누적.
   변경 이력 자체가 ①시계열 데이터가 된다.
7. **비교표 + 단위 통일** — 하우스 룰(비교표 필수, JSON 코드블록 금지)에 더해,
   모든 수치를 동일 단위·동일 조건으로 정규화 ($/1M tokens 등). 각주로 예외 표기.
8. **출처 표기** — 외부 수치는 전부 출처+날짜. 자체 실측 수치는
   "measured by jsonhouse, YYYY-MM-DD"로 명시해 우리가 원출처임을 선언.

### 데이터 파일과의 정합

- `_data/YYYY-MM-DD-{slug}.json`의 `key_facts`에는 이 포스트가 1차 출처인
  수치를 우선 배치하고, `source`에 `"jsonhouse original measurement, YYYY-MM-DD"`
  형식으로 기록한다 (외부 인용 수치와 명확히 구분).
- 시계열 원본은 `_data/pricing_history/` 등 비공개 하위 디렉터리에 보관하고,
  포스트에는 가공본만 노출한다 (원본 시계열은 Phase 3 자산).

## 3. 주목받는 1차 포스트를 만드는 방법 (배포·증폭)

만드는 것의 절반은 발견되게 하는 것이다. 발행 시 아래를 실행한다.

### 타이밍 (가장 큰 지렛대)

- **이벤트 후 48시간 안에 발행** — 모델 출시·가격 변경 직후가 검색량 피크이면서
  경쟁 콘텐츠가 가장 얇은 구간이다. 신모델 비교·신가격 반영 포스트는 속도가 품질만큼 중요.
- 이벤트가 예고되면(출시 예정 등) 골격을 미리 준비하고 수치만 채워 발행한다.

### 검색·AEO (사람과 에이전트가 찾게)

- 제목·H2를 실제 질문 형태로 (SEO Rules의 TL;DR·FAQ·150단어 내 답변 규칙은 기본)
- 핵심 수치는 표 + 한 문장 요약 병기 — 답변 엔진은 문장을 인용하고 표로 검증한다
- 기계 가독성 3종 확인: `/data/{slug}.json` 200 응답, llms.txt 데이터셋 목록 등재,
  JSON-LD 정상 생성

### 재방문 신호 (크롤러가 다시 오게)

- 갱신 주기를 지키고, 갱신 시 `last_modified_at`·`data_updated`·Changelog를
  함께 올린다 — "살아있는 데이터"라는 신호가 재크롤과 재인용을 만든다
- 갱신이 누적된 포스트가 신규 포스트보다 인용 확률이 높다. 새 글 늘리기보다
  기존 데이터 포스트를 살아있게 유지하는 것이 우선

### 커뮤니티 시딩 (초기 발견)

- HN / Reddit / dev.to에서 **관련 질문에 데이터로 답하는 형태**로만 공유한다
  (링크 투척 금지 — 표의 핵심 수치를 답변에 직접 쓰고 출처로 링크)
- 데이터 자체가 뉴스일 때만 직접 제출 (예: "6개월 가격 추적 결과 평균 X% 하락")
- 실측(②) 포스트는 방법론 비판을 환영한다고 명시 — 논쟁이 배포다

### 인용 루프 (관찰을 다시 콘텐츠로)

- 주간 인용 추적(`_data/citation_log/`)으로 어떤 포스트·수치가 인용되는지 기록
- 봇 로그에서 어떤 데이터 파일이 긁히는지 확인 → 수요가 확인된 주제를 증설
- 인용·크롤링 관찰 자체를 ③독점 관찰 포스트로 재생산한다 (루프 완성)

## 기획 승인 요청 형식

데이터형 포스트를 기획하면 착수 전에 아래를 한국어로 보고한다:

- 주제 + 어떤 경로(①~④)로 1차 출처가 되는가
- 판별 테스트 통과 논리 (AI가 왜 우리를 건너뛸 수 없는가)
- 갱신 주기 약속과 유지 방법
- 경쟁 데이터셋과 우리가 이기는 축
